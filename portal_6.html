Data Tables

Gen3 - CDE Data Tables

Comprehensive documentation of the data structure and fields within the CDNM Gen3 system. Explore the data model that powers your research.

Study

The study table is shared between CDE and SGG studies. This table has a unique identifier for each study and some metadata and descriptions for each.

For CDE the study table has the 4 studies and their descriptions:

NHS1

Nurses' Health Study 1

NHS2

Nurses' Health Study 2

NHS3

Nurses' Health Study 3

GUTS

Growing Up Today Study

Subject

The subject table is shared between CDE and SGG subjects. This table has a unique identifier for each subject (study participant) and metadata associated with them.

Fields of Interest for CDE Data

subject_id

This is the participants 'Cohort ID' (their regular Nurses' ID including check-digit). For example: '123456A'

alias_id

'External ID' or 'fake' ID used for external use of Nurses' data. These are 10 character IDs that start with 'X'.

enrollment_date

Participant enrollment date or study start date:

  • NHS1 – 06/15/1976
  • NHS2 – 06/15/1989
  • GUTS1 – 06/15/1996
  • GUTS2 – 06/15/2004
  • NHS3 – 15th of the Month and Year that they responded to Mod1

active_status

'active' or 'inactive' value depending on the participants current Name&Address status value.

For NHS1, NHS2, and GUTS these are set to 'active' when they have any of the following status codes:

  • 0.address changed
  • 1.completed questionnaire
  • 5.telephone only
  • 6.short form only
  • 7.alive in this year (nhs1 & guts) / 7.online short only (nhs2)

Any other status is considered 'inactive' (2.refusal, 3.dead, etc.)

For NHS3 any status code in the Name & Address file is considered 'inactive' (M.max mod1 reminders, D.duplicate, etc.)

Note: This indicates if the subject is actively participating in the study (or is eligible to receive more questionnaires). This does NOT indicate how recently they participated. We may not have heard from them for many years but they are still 'active'.

opt_out_status

Indicates that the person has an opt-out status of some kind in the opt-out file: 'all', 'dna', 'old', or 'sub'.

Should be reflected in DER files (as either no record in DER file OR missing DOB in DER file). Non-missing opt_out_status indicates that for that type of data ('all', 'dna', 'sub') this person should be excluded from analysis.

Important: We want to save their data but observations from this subject should not be included in analysis.

race

Race categories following the federal standard (including 'multi-racial' and 'other' categories).

  • NHS1 and NHS2: From derived files – race9204 (NHS1; der7620) – race8905 (NHS2; der8919)
  • GUTS: Similar algorithm using combination of baseline and 2014 race reports, with participants mothers race used to fill missing responses
  • NHS3: Using mod1 responses, combined using same precedence as NHS1/2

'Multi-racial' is assigned when more than one of the race fields is marked. 'Other/unknown' applies to 'Other' responses (NHS3 only) as well as no race response on a completed questionnaire.

ethnicity

Self reported 'Hispanic' or 'Not Hispanic' values from a combination of questionnaires for participant. This value comes from the same sources as the race field.

sex

This is meant to represent the participants natal sex or gender assigned at birth.

  • NHS1 & NHS2: Simply set to Female (any participants later found to be male have been removed from the cohort)
  • GUTS: From the natal_sex field in gutsder9623
  • NHS3: From gender_m1 field in mod1 – "What sex were you assigned at birth, on your original birth certificate?" If missing, Name & Address field 'gender' is used

Instrument

An instrument can be thought of as a dataset belonging to a study. This table is CDE specific and contains a unique identifier and metadata for any CDE dataset that has been loaded into Gen3.

Fields of Interest

instrument_id

This is a unique identifier for the dataset/instrument. This is mostly set to the SAS macro name.

Examples:

  • 'nur12'
  • 'der8917'

Except for instances where the SAS macro has more than one cycle: 'n76' vs 'n767880'

instrument_name

This is an alternate name for the dataset allowing duplicates. These should match the corresponding source SAS macro name.

Examples:

  • 'NUR12'
  • 'DER8917'
  • 'N767880'

instrument_description

A description of the dataset/instrument.

Examples:

  • 'NHS1 1998 Questionnaire'
  • 'NHS1 1980 raw nutrient intakes'

instrument_year

This is the cycle year or mod associated with the instrument:

  • For 'NUR' and nutrient data this is set to the 4 digit year of the associated cycle
  • For CDX, DER and NHS3 MOD data this is set to N/A
  • For NHS3 we don't really have a cycle year so it's blank in gen3

instrument_source

A path and/or a more descriptive explanation of the data source. This won't be visible in the explorer.

Instrument-Field (Variables)

The Instrument-Fields table is a collection of fields or variables associated with an instrument. Fields or variables in this table are unique but can occur in more than one instrument. This table is specific to CDE.

Fields of Interest

field_id (Variable ID)

Unique identifier for each field or variable. This is typically set to the 'stem' name or the non-cycle portion of the variable name in the read macro.

Example: 'wt89' would be set to 'wt'

These should be harmonized across cycles so that the same field/variable is associated with more than one dataset/instrument. For example, the same 'wt' field_id will be associated with all 'wt' fields (wt89, wt91, wt93, etc.)

field_name

I believe this redundant, I don't think we're using it. Same as field_id.

field_description (Variable Description)

A description of what the variable is observing. This is either a harmonized description of the field or simply the variable description from the data dictionary (as-is).

Example: "smoking, cigarettes per day"

In some cases this is the full Question as it appeared on the questionnaire.

Observations

The observations table in our Gen3 data model is a simplified version of the observations table in the OMOP Common Data Model. This allows us to load any 2 dimensional dataset into Gen3. There is one record for every value-date-field combination (This allowed us to reduce the number of records from billions to millions).

Field Information

field_id

This is the field of which the observation is from. This is the same unique identifier field_id from the instrument-field table.

observation_date

This is the date of the observation. All dates are set to the 15th of the month and year. This might be updated to 'months since enrollment' (integer values) in the future.

Depending on the source this can be defined in several ways:

Main questionnaires

For observations from a questionnaire (nur15, mod1 etc.) this is the date of response for the questionnaire.

Derived files

For derived file observations (der8919, der7620) this date is set to June 15th of the cycle year associated with the field_id observation.

Examples:

  • bmi89 – 06/15/1989
  • bmi91 – 06/15/1991

If the source field isn't associated with a cycle ('BMI at 18' for example) the date defaults to the participants enrollment date.

Nutrient data

For observations from a data file that is derived for a whole cycle (n84_nts, n84_ant) the date is set to June 15th of the cycle year for all observations in that file.

Example: All observation_dates from n20_atd are set to 06/15/2020

Disease followup data

Observations from the disease files (nhs1_cdx_br_cases) are set to the diagnosis date (dxmonth) for the record.

observation_text

The observation value – numeric or character values.

Formatted or categorical values are used when available rather than raw values.

Example: 1 vs 'yes' – observation_text will be 'yes'

source_instrument

Instrument or dataset where the value for field_id was observed. This is a string value and corresponds to the instrument_id values.

subjects

List of subject_ids that responded to field_id with observation_text on observation_date. This will be displayed as a count of subject_ids in the explorer.

Key Insight: This structure allows for efficient querying and analysis of longitudinal data across multiple studies and timepoints, while maintaining data integrity and relationships.

Data

FAIR Data Principles

The Gen3 platform is built around the FAIR principles — making research data Findable, Accessible, Interoperable, and Reusable.

What is FAIR?

Findable

Data is assigned globally unique identifiers with rich metadata for discovery.

Accessible

Data is retrievable using standardized, open protocols.

Interoperable

Data uses formal, shared vocabularies and standards.

Reusable

Data has clear usage licenses, detailed provenance, and meets community standards.

Data Commons Model

A data commons provides researchers with a shared, governed environment to co-locate data with compute and tools. Gen3 powers several major NIH data commons.

API

API Reference

Complete reference for Gen3 platform APIs. All endpoints require authentication.

Authentication

Getting an Access Token

Navigate to your Profile and click "Create API Key". Download the credentials.json file.

Requesting a Token

curl -X POST https://data-portal.example.com/user/credentials/api/access_token \
  -H "Content-Type: application/json" \
  -d @credentials.json

Using the Token

curl -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  https://data-portal.example.com/api/v0/submission/

Core Endpoints

Submission API

/api/v0/submission/ — CRUD for data records.

Index API

/index/ — File lookup by GUID or hash.

Query API

/api/v0/submission/graphql — GraphQL queries.

Export API

/api/v0/submission/export — Bulk data export.

MDS (Metadata)

/mds/metadata/ — Dataset metadata.

Fence (Auth)

GraphQL Queries

Basic Query

{
  subject(first: 10) {
    submitter_id
    project_id
    vital_status
  }
}

Filtered Query

{
  subject(first: 50, filter: { vital_status: "Alive" }) {
    submitter_id
    project_id
    age_at_enrollment
  }
}

Aggregation Query

{
  _aggregation {
    subject {
      _totalCount
      vital_status { histogram { key count } }
    }
  }
}
Tip: Use the Query Builder to visually construct queries, then copy the GraphQL.
Schema

Data Model

Schema Overview

The data model is a directed acyclic graph (DAG) of nodes connected by relationships.

Key Concepts

Program — Top-level organizational unit.

Project — A collection of data within a program.

Node — A data entity with a defined schema.

Edge — A directional relationship between nodes.

Common Node Types

Subject

Research participant with demographics and enrollment info.

Sample

Biological specimen linked to a subject.

Diagnosis

Clinical diagnosis with disease type and classification.

Treatment

Treatment or intervention record.

Lab Result

Laboratory test result with values and units.

Data File

Reference to a stored file (genomic, imaging, etc.).

Relationships

Typical Submission Order

Program → Project → Subject → Sample → Lab Result
                                    ↘ Diagnosis
                                    ↘ Treatment
                                    ↘ Data File

Each child node references its parent via a submitter_id link.

Prepare Your Data

01
Review the Schema

02
Format Your Data

Structure as JSON or TSV with matching column headers.

03
Include Parent Links

Validate & Submit

Via the API

curl -X PUT https://data-portal.example.com/api/v0/submission/program/project/ \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d @your_data.json
Dry run first: Append ?dry_run=true to validate without submitting.

Pipeline Stages

Validation → Indexing → Transformation → Availability (15–60 min typical)

Check status: Use the System Status page. If data doesn't appear after 2 hours, submit a ticket.
